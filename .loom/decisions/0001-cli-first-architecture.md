# CLI 우선 아키텍처

## 맥락

aeira는 loom/shulker 스킬의 문서 탐색 백엔드로 동작해야 한다. 인터페이스 형태를 MCP 서버로 할지, CLI 도구로 할지 결정이 필요했다.

## 결정

CLI를 핵심 인터페이스로 한다. 스킬은 Bash를 통해 CLI를 직접 호출한다. MCP 어댑터는 필요 시 CLI 위에 얹는 선택적 레이어로 둔다.

### 제약 조건

- Obsidian과 달리 전용 앱에 종속되지 않아야 한다: GUI 없이 동작해야 함
- 독립적인 백엔드로 존재해야 한다: 특정 통합 방식에 종속되지 않아야 함

### 전제 조건

- 스킬에서 Bash 도구로 CLI 실행이 충분히 빠르다: 디스크 캐시가 있으므로 매 호출의 오버헤드는 인덱스 로드 비용에 한정
- CLI 위에 MCP 레이어를 얹는 것은 역방향보다 쉽다: 옵션 가치가 높은 선택

## 대안

### MCP 서버 우선

- 설명: MCP 서버로 구현하고, 서버가 프로세스로 상주하며 인메모리 그래프를 유지
- 트레이드오프: 인메모리 유지로 조회가 빠르지만, 서버 설정/관리 오버헤드가 생기고 MCP 클라이언트에 종속됨

### 라이브러리 우선

- 설명: 코어를 라이브러리로 만들고, CLI와 MCP를 어댑터로 구현
- 트레이드오프: 가장 유연하지만, 초기 프로젝트에서 레이어 분리는 과도한 추상화
