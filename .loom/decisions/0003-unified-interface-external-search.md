# 통합 인터페이스 + ir 검색 위임

## 맥락

aeira의 소비자(loom/shulker)는 세 가지 수준의 조회가 필요하다: 그래프 순회, 문서 이름 탐색, 내용 검색. 이를 별도 도구로 분리할지, 하나의 도구로 통합할지 결정이 필요했다.

## 결정

aeira가 통합 인터페이스를 제공한다. 내용 검색은 ir(로컬 시맨틱 검색 엔진)에 위임하고, 그래프 탐색은 내장 기능으로 구현한다. aeira는 ir를 내부적으로 오케스트레이션하며, 소비자는 ir의 존재를 알 필요 없다.

### 제약 조건

- 소비자의 사용 패턴이 "검색으로 진입 → 그래프로 탐색"의 조합: 분리하면 소비자마다 두 도구를 조합하는 부담이 발생
- 별도 도구로 분리하면 문서 식별자 불일치 위험: 검색 결과와 그래프 노드 간 매핑이 깨질 수 있음

### 전제 조건

- ir가 BM25 + 벡터 + 리랭킹 하이브리드 검색, 한국어 지원, per-collection SQLite를 제공: 자체 구현 대비 큰 이점
- ir의 collection 모델이 aeira의 source 개념과 1:1 대응: 통합이 자연스러움

## 대안

### 검색 인덱스 내장

- 설명: aeira가 자체 전문 검색 인덱스를 구축하고 관리
- 트레이드오프: 완전한 통제가 가능하지만, 이미 풀린 문제를 다시 풀게 됨. 인덱스 관리 복잡도 증가

### 완전 분리

- 설명: 검색 도구와 그래프 도구를 독립적으로 구축
- 트레이드오프: 각 도구가 단순해지지만, 소비자에게 통합 부담 전가. 문서 식별자 불일치 위험. 복합 질의 불가

### 단순 grep 위임

- 설명: ripgrep 등 텍스트 검색 도구에 위임
- 트레이드오프: 의존성이 가볍지만, 시맨틱 검색·한국어 형태소 분석 등 ir가 제공하는 기능을 포기
