# 프리미티브 기반 쿼리 인터페이스

## 맥락

그래프 쿼리의 인터페이스를 소비자 시나리오에 맞춘 고수준 명령(`related-concepts`, `backlinks`, `overview`)으로 할지, 저수준 프리미티브로 할지 결정이 필요했다. 소비자는 loom(shape/review/calibrate)과 shulker(recall)이며, 모두 "진입점 → 확장" 패턴을 공유한다.

## 결정

저수준 프리미티브 세 개를 제공하고, 소비자가 조합하게 한다.

- **neighbors(node, direction?)** — 1-hop 이웃 반환. 대부분의 탐색 시나리오를 커버
- **path(from, to)** — 두 노드 간 모든 경로 반환. 중간 노드를 포함하여 N-hop 상호작용을 드러냄
- **snapshot()** — 전체 노드 + 엣지 목록. 전체 구조 파악용

### 제약 조건

- 소비자가 LLM 스킬이므로 결과 조합 능력이 있음: 고수준 추상화의 이점이 적음
- 시나리오가 추가되어도 프리미티브가 변하지 않아야 함: 소비자별 명령을 만들면 소비자 추가마다 aeira를 수정해야 함

### 전제 조건

- 소비자의 쿼리 패턴이 세 프리미티브로 환원 가능: 현재 식별된 모든 시나리오가 neighbors/path/graph의 조합으로 표현됨
- path에서 순환 참조 방어와 경로 수 상한이 필요: 노드 재방문 금지로 사이클을 차단하고, 결과 수를 제한

## 대안

### 시나리오 기반 고수준 명령

- 설명: `related-concepts`, `backlinks`, `overview` 등 소비자의 의도에 맞춘 명령을 제공
- 트레이드오프: 호출이 직관적이지만, 소비자가 추가될 때마다 aeira에 명령을 추가해야 함. 프리미티브의 조합으로 표현 가능한 것을 고정된 시나리오로 묶어버림

## 후기

- 세 프리미티브가 Graph 자료구조를 입력받는 순수 함수로 구현됨. 소비자는 store.loadEdges()로 Graph를 얻은 뒤 자유롭게 조합 가능
- path의 DFS + 방문 집합 방식이 순환 방어와 경로 수 상한을 자연스럽게 해결함
