# aeira 코어 구현

## 핵심 구조

데이터 흐름은 단방향이다:

```
[source 경로] → 문서 스캔 → 위키링크 파싱 → 그래프 구축 → 인덱스 저장
                                                              ↓
[CLI 쿼리]   →                          인덱스 로드 → 그래프 조회 → 결과 출력
```

구축과 조회는 분리된 동작이다.

인덱스 경로 매핑: source 경로의 해시로 프로젝트 외부 캐시 위치를 결정론적으로 도출한다.

## 작업 목록

### 0. 프로젝트 세팅

#### 작업 견적

- 기술 스택 결정: 보통 — 언어/런타임 선택이 전체 구현에 영향
- 초기 구조: 낮음 — 기계적 작업

#### 작업 순서

- 의존: 없음 (최초 태스크)
- 병렬: 없음

#### 세부 작업 목록

- [ ] 기술 스택: TypeScript + Node.js
- [ ] 패키지 초기화, 빌드/테스트 설정
- [ ] CLI 엔트리포인트 스캐폴딩

### 1. 위키링크 파서

#### 작업 견적

- 파싱 로직: 낮음 — 정규식 기반, `[[target]]`과 `[[target|alias]]` 두 패턴
- 엣지 케이스: 보통 — 코드블록/인라인코드 내부의 위키링크 무시, 중첩 괄호 등

#### 작업 순서

- 의존: 0
- 병렬: 2

#### 세부 작업 목록

- [ ] 위키링크 파싱 함수 — 마크다운 텍스트 → `{ target, alias? }[]` 반환
- [ ] 코드블록/인라인코드 내부 위키링크 제외
- [ ] 파서 테스트

### 2. 문서 스캐너

#### 작업 견적

- 파일 탐색: 낮음 — 재귀 glob, 기존 라이브러리 활용
- 경로 정규화: 보통 — 상대/절대 경로 통일, 문서명 식별 규칙

#### 작업 순서

- 의존: 0
- 병렬: 1

#### 세부 작업 목록

- [ ] source 경로 하위 `.md` 파일 재귀 탐색
- [ ] 경로 정규화 — 문서 식별자로 사용할 이름 도출 (파일명 stem 또는 상대 경로)
- [ ] 스캐너 테스트

### 3. 그래프 구축

#### 작업 견적

- 자료구조: 보통 — adjacency list 기반 방향 그래프, in/out 양방향 조회 지원
- 빌더 조합: 낮음 — 스캐너 + 파서를 순회하며 노드/엣지 추가

#### 작업 순서

- 의존: 1, 2
- 병렬: 없음

#### 세부 작업 목록

- [ ] 그래프 자료구조 — 노드 집합, outgoing/incoming adjacency list
- [ ] 빌더 — 문서 목록을 순회하며 파싱 → 노드/엣지 생성
- [ ] dangling link 처리 — 존재하지 않는 문서를 가리키는 위키링크의 처리 방식 결정
- [ ] 그래프 구축 테스트

- 직렬화: 낮음 — JSON 또는 바이너리, 초기에는 단순한 포맷으로 시작
- 변경 감지: 보통 — 파일 mtime 기반 유효성 판단, 증분 갱신 전략

#### 작업 순서

- [ ] 직렬화/역직렬화 — 그래프 ↔ 디스크 포맷
- [ ] 변경 감지 — 파일 mtime 저장, 변경된 문서만 재파싱

## 검증

- 인덱스 구축 시 프로젝트 외부에 인덱스가 생성된다
- 대상 프로젝트 내부에는 아무 파일도 생기지 않는다
- 문서 변경 후 재구축하면 변경분만 갱신된다
- `.loom/` 디렉토리를 source로 지정하여 loom 문서에서 동작함을 확인한다
